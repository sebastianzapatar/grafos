¿Qué es un Grafo?

Estructura de datos fundamental en ciencias de la computación

📐 Definición Formal

Un grafo G = (V, E) es un par ordenado donde:

  • V = conjunto finito de vértices (nodos)
  • E ⊆ V × V = conjunto de aristas (conexiones)

📖 Terminología Clave

  • Grado: número de aristas que inciden en un nodo
  • Camino: secuencia de vértices conectados por aristas
  • Ciclo: camino que inicia y termina en el mismo nodo
  • Grafo conexo: existe camino entre todo par de nodos
  • DAG: Directed Acyclic Graph (clave en compiladores)

➡️ Grafo Dirigido

Las aristas tienen dirección: (u, v) ≠ (v, u)

Ejemplo: flujo de control en un compilador

↔️ Grafo No Dirigido

Las aristas no tienen dirección: {u, v} = {v, u}

Ejemplo: red social de amistades

⚖️ Grafo Ponderado

Cada arista tiene un peso/costo asociado: w(u,v)

Ejemplo: mapa de rutas con distancias

Representaciones de Grafos

¿Cómo almacenar un grafo en memoria?

📊 Matriz de Adyacencia

Matriz V×V donde M[i][j] = 1 si existe arista de i a j

Representación // Grafo: 0→1, 0→2, 1→3, 2→3 // 0 1 2 3 // 0 [ 0, 1, 1, 0 ] // 1 [ 0, 0, 0, 1 ] // 2 [ 0, 0, 0, 1 ] // 3 [ 0, 0, 0, 0 ] int[][] matrix = new int[V][V]; matrix[0][1] = 1; matrix[0][2] = 1;
  • ✅ Acceso O(1) para verificar arista
  • ✅ Ideal para grafos densos
  • ❌ Espacio O(V²) siempre
  • ❌ Ineficiente para grafos dispersos

📋 Lista de Adyacencia

Arreglo de listas: adj[i] contiene los vecinos de i

Representación // Grafo: 0→1, 0→2, 1→3, 2→3 // 0: [1, 2] // 1: [3] // 2: [3] // 3: [] List<List<Integer>> adj = new ArrayList<>(); adj.get(0).add(1); adj.get(0).add(2);
  • ✅ Espacio O(V + E) eficiente
  • ✅ Ideal para grafos dispersos
  • ✅ Iterar vecinos es O(grado)
  • ❌ Verificar arista O(grado)

BFS — Búsqueda en Anchura

Explora nivel por nivel usando una Cola (FIFO) · Complejidad: O(V + E)

Listo
BFS(grafo, inicio): cola ← [inicio] visitado[inicio] ← true mientras cola no vacía: nodo ← cola.dequeue() para cada vecino de nodo: si no visitado[vecino]: visitado[vecino] ← true cola.enqueue(vecino)
Java public void bfs(int start) { // Array de visitados: evita ciclos boolean[] visited = new boolean[V]; // Distancia por aristas desde start int[] distance = new int[V]; // Cola FIFO: procesa nivel por nivel Queue<Integer> queue = new LinkedList<>(); // Inicializar nodo origen visited[start] = true; distance[start] = 0; queue.add(start); while (!queue.isEmpty()) { // Extraer frente de la cola int node = queue.poll(); // Explorar vecinos for (int nb : adj.get(node)) { if (!visited[nb]) { visited[nb] = true; distance[nb] = distance[node] + 1; queue.add(nb); } } } }

Dijkstra — Camino Más Corto

Algoritmo greedy con PriorityQueue · O((V+E) log V) · Solo pesos ≥ 0

Listo
Dijkstra(grafo, inicio): dist[*] ← ∞; dist[inicio] ← 0 pq ← MinHeap con (inicio, 0) mientras pq no vacía: (u, d) ← pq.extractMin() si d > dist[u]: continuar para cada arista (u→v, peso w): si dist[u] + w < dist[v]: // Relajar dist[v] ← dist[u] + w pq.insert(v, dist[v])
Java public void dijkstra(int start) { int[] dist = new int[V]; Arrays.fill(dist, Integer.MAX_VALUE); dist[start] = 0; // Min-heap: ordena por distancia PriorityQueue<int[]> pq = new PriorityQueue<>( Comparator.comparingInt(a -> a[1]) ); pq.add(new int[]{start, 0}); while (!pq.isEmpty()) { int[] curr = pq.poll(); int u = curr[0], d = curr[1]; // Obsoleto si ya hay mejor camino if (d > dist[u]) continue; // RELAJACIÓN de aristas for (Edge e : adj.get(u)) { if (dist[u] + e.weight < dist[e.target]) { dist[e.target] = dist[u] + e.weight; pq.add(new int[]{ e.target, dist[e.target] }); } } } }

Bellman-Ford

Soporta pesos negativos · Detecta ciclos negativos · O(V × E)

Listo
BellmanFord(grafo, inicio): dist[*] ← ∞; dist[inicio] ← 0 repetir V-1 veces: para cada arista (u→v, peso w): si dist[u] + w < dist[v]: dist[v] ← dist[u] + w // Detectar ciclos negativos: para cada arista (u→v, peso w): si dist[u] + w < dist[v]: error "Ciclo negativo"
Java public void bellmanFord(int start) { int[] dist = new int[V]; Arrays.fill(dist, Integer.MAX_VALUE); dist[start] = 0; // V-1 iteraciones: el camino más // largo sin ciclos tiene V-1 aristas for (int i = 1; i < V; i++) { // Relajar TODAS las aristas for (Edge e : edges) { if (dist[e.source] != MAX && dist[e.source] + e.weight < dist[e.target]) { // ¡Camino más corto! dist[e.target] = dist[e.source] + e.weight; } } } // Iteración extra = ciclo negativo for (Edge e : edges) { if (dist[e.source] + e.weight < dist[e.target]) { System.out.println( "¡Ciclo negativo!"); } } }

Floyd-Warshall — Todos los Pares

Programación dinámica · Distancia mínima entre TODOS los pares · O(V³)

Listo
FloydWarshall(grafo): dist[][] ← matriz de pesos del grafo // k = vértice intermedio para k ← 0 hasta V-1: para i ← 0 hasta V-1: para j ← 0 hasta V-1: si dist[i][k] + dist[k][j] < dist[i][j]: dist[i][j] ← dist[i][k] + dist[k][j] // Si dist[i][i] < 0 → ciclo negativo

💡 Idea Central

Para cada par (i, j), ¿conviene pasar por un nodo intermedio k?

dist[i][j] = min( dist[i][j], // camino directo dist[i][k] + dist[k][j] // vía k )
Java // Triple bucle: k DEBE ser externo for (int k = 0; k < V; k++) { for (int i = 0; i < V; i++) { for (int j = 0; j < V; j++) { if (dist[i][k] != INF && dist[k][j] != INF && dist[i][k] + dist[k][j] < dist[i][j]) { dist[i][j] = dist[i][k] + dist[k][j]; next[i][j] = next[i][k]; } } } }

¿Qué es un Ciclo?

Camino que inicia y termina en el mismo vértice · Detección con DFS · O(V + E)

Listo

🎨 DFS con 3 colores (grafos dirigidos)

⚪ BLANCO: sin visitar · 🩶 GRIS: en la pila de recursión · ⚫ NEGRO: terminado

Si una arista apunta a un nodo GRIS → volvimos a un nodo de nuestro propio camino → ¡hay ciclo! (back edge). Un grafo dirigido sin ciclos es un DAG.

En grafos no dirigidos: hay ciclo si un vecino ya visitado NO es el padre del nodo actual.

Java — DeteccionCiclos.java // BLANCO=0, GRIS=1, NEGRO=2 private boolean dfsCiclo(int u, int[] color) { color[u] = GRIS; // entra a la recursión for (int v : adj.get(u)) { if (color[v] == GRIS) { // ¡Back edge! v está en // nuestro camino actual return true; // CICLO } if (color[v] == BLANCO) { parent[v] = u; if (dfsCiclo(v, color)) return true; } // NEGRO: ya explorado, ignorar } color[u] = NEGRO; // sale de la recursión return false; }

🤔 ¿Por qué no basta "visitado sí/no"?

En el DAG 0→1→3 y 0→2→3, el nodo 3 se alcanza por dos caminos sin haber ciclo. El GRIS distingue "está en MI camino" de "ya explorado por otro lado".

Ciclos Negativos

Ciclo cuya suma de pesos es < 0 (ej: 3 + 2 + (−7) = −2) · El camino más corto deja de existir

Listo

💥 ¿Por qué son un problema?

Cada vuelta al ciclo reduce el costo total: la "distancia mínima" tiende a −∞ y el camino más corto no está definido. Por eso Dijkstra no los soporta.

🔍 Detección: Bellman-Ford con una iteración extra (si algo aún se relaja → ciclo negativo), o Floyd-Warshall si dist[i][i] < 0.

Java — CicloNegativo.java (reconstrucción) int[] dist = new int[V]; // en 0: busca en TODO el grafo int[] parent = new int[V]; int x = -1; // último nodo relajado // V iteraciones (una MÁS que Bellman-Ford) for (int i = 0; i < V; i++) { x = -1; for (Edge e : edges) { if (dist[e.source] + e.weight < dist[e.target]) { dist[e.target] = dist[e.source] + e.weight; parent[e.target] = e.source; x = e.target; // hubo relajación } } } // ¿La iteración V aún relajó? → ciclo negativo if (x == -1) return sinCiclo; // Retroceder V veces para caer DENTRO del ciclo for (int i = 0; i < V; i++) x = parent[x]; // Seguir parent[] hasta cerrar el ciclo for (int v = x; ; v = parent[v]) { ciclo.add(v); if (v == x && ciclo.size() > 1) break; }

Kahn — Orden Topológico

Resuelve dependencias en un DAG usando grados de entrada (In-Degree) y una Cola · O(V + E)

Listo
Kahn(DAG): inDegree[*] ← 0 para cada arista (u→v): inDegree[v]++ cola ← {nodos con inDegree == 0} mientras cola no vacía: u ← cola.dequeue() procesar(u) para cada vecino v de u: inDegree[v]-- si inDegree[v] == 0: cola.enqueue(v)
Java public void topologicalSort() { int[] inDegree = new int[V]; for (int i = 0; i < V; i++) { for (int neighbor : adj.get(i)) { inDegree[neighbor]++; } } Queue<Integer> queue = new LinkedList<>(); for (int i = 0; i < V; i++) { if (inDegree[i] == 0) queue.add(i); } List<Integer> topOrder = new ArrayList<>(); while (!queue.isEmpty()) { int node = queue.poll(); topOrder.add(node); for (int neighbor : adj.get(node)) { inDegree[neighbor]--; if (inDegree[neighbor] == 0) { queue.add(neighbor); } } } }

Union-Find — Union y Union by Rank

DSU (Disjoint Set Union): la estructura que hace eficiente a Kruskal · find() + union()

🔗 ¿Qué es Union-Find?

Mantiene conjuntos disjuntos. Dos operaciones:

  • find(x): ¿a qué conjunto pertenece x? (devuelve la raíz)
  • union(x,y): fusionar los conjuntos de x e y

En Kruskal: si find(u) == find(v), agregar la arista u—v formaría un ciclo.

⚡ Path Compression (find)

Al buscar la raíz, colgamos cada nodo directamente de ella:

Antes: 0←1←2←3 (cadena) find(3): recorre 3→2→1→0 Después: 0←1, 0←2, 0←3 (plano)

⚖️ Union Naive vs Union by Rank

Union Naive Union by Rank
Estrategia Cuelga rx de ry (arbitrario) Cuelga el más bajo del más alto
Peor caso O(n) sin PC O(log n) sin PC
Con PC O(log n) amort. O(α(n)) ≈ O(1)

PC = Path Compression · α = inversa de Ackermann

Java — Union Naive // Union NAIVE: cuelga rx de ry sin // considerar cuál árbol es más alto boolean union(int x, int y) { int rx = find(x); int ry = find(y); if (rx == ry) return false; // Siempre cuelga rx de ry // (puede crear cadenas largas) parent[rx] = ry; return true; }
Java — Union by Rank // UNION BY RANK: el árbol más bajo // se cuelga del más alto → O(α(n)) boolean union(int x, int y) { int rx = find(x); int ry = find(y); if (rx == ry) return false; if (rank[rx] < rank[ry]) { parent[rx] = ry; // rx más bajo } else if (rank[rx] > rank[ry]) { parent[ry] = rx; // ry más bajo } else { parent[ry] = rx; // misma h. rank[rx]++; // crece en 1 } return true; }
Java — find (Path Compression) // find con PATH COMPRESSION // Cuelga cada nodo directo de la raíz int find(int x) { if (parent[x] != x) parent[x] = find(parent[x]); return parent[x]; }

Kruskal — MST

Greedy por Aristas. Usa Union-Find (Disjoint Set) para evitar ciclos · O(E log E)

Listo
Kruskal(grafo): ordenar aristas por peso ascendente mst ← [] uf ← UnionFind(V) para cada arista (u-v, peso) en orden: si uf.find(u) ≠ uf.find(v): uf.union(u, v) mst.add(arista) si mst.tamaño() == V-1: romper retornar mst
Java public void kruskal() { // 1. Ordenar aristas por peso ascendente Collections.sort(edges); UnionFind uf = new UnionFind(V); List<Edge> mst = new ArrayList<>(); int costoTotal = 0; // 2. Procesar cada arista for (Edge e : edges) { // union() devuelve true si u y v // estaban en conjuntos diferentes if (uf.union(e.u, e.v)) { mst.add(e); costoTotal += e.weight; // 3. Parar al tener V-1 aristas if (mst.size() == V - 1) break; } } }

Prim — MST

Greedy por Nodos. Hace crecer un único árbol usando PriorityQueue · O((V+E) log V)

Listo
Prim(grafo, inicio): minCost[*] ← ∞; minCost[inicio] ← 0 pq ← MinHeap con (inicio, 0) mientras pq no vacía: u ← pq.extractMin() si enArbol[u]: continuar enArbol[u] ← true para cada vecino v de u con peso w: si no enArbol[v] y w < minCost[v]: minCost[v] ← w padre[v] ← u pq.insert(v, w)
Java public void prim(int start) { boolean[] enArbol = new boolean[V]; int[] minCost = new int[V]; int[] parent = new int[V]; Arrays.fill(minCost, Integer.MAX_VALUE); Arrays.fill(parent, -1); minCost[start] = 0; PriorityQueue<int[]> pq = new PriorityQueue<>( Comparator.comparingInt(a -> a[1])); pq.add(new int[]{start, 0}); while (!pq.isEmpty()) { int u = pq.poll()[0]; if (enArbol[u]) continue; enArbol[u] = true; for (Edge e : adj.get(u)) { if (!enArbol[e.target] && e.weight < minCost[e.target]) { minCost[e.target] = e.weight; parent[e.target] = u; pq.add(new int[]{e.target, e.weight}); } } } }

Algoritmo Tipo Complejidad Pesos Negativos Caso de Uso
BFS Recorrido O(V + E) N/A Camino más corto sin pesos, componentes conexas
DFS Recorrido O(V + E) N/A Ciclos, orden topológico, componentes fuertes
Dijkstra Greedy O((V+E) log V) ❌ No Camino más corto desde un origen (pesos ≥ 0)

Floyd-Warshall DP O(V³) ✅ Sí Todos los pares, grafos pequeños/medianos

Ejercicio 1 — BFS

★★★★

Recorrido BFS: Determinar el orden de visita

Dado el siguiente grafo dirigido, determina el orden de visita BFS comenzando desde el nodo 0:

Nodos: {0, 1, 2, 3, 4, 5} Aristas: 0→1, 0→2, 1→3, 1→4, 2→4, 2→5, 3→5, 4→5 Representación visual: 0 / \ 1 2 / \ / \ 3 4 5 \ | / → 5 ←

Pregunta: ¿Cuál es el orden de visita BFS? ¿Cuál es la distancia (en aristas) del nodo 0 al nodo 5?

✅ Solución

Orden BFS: 0, 1, 2, 3, 4, 5

Explicación paso a paso:

  • Inicio: Cola = [0], Visitados = {0}
  • Sacar 0 → vecinos 1, 2 → Cola = [1, 2], Visitados = {0, 1, 2}
  • Sacar 1 → vecinos 3, 4 → Cola = [2, 3, 4], Visitados = {0, 1, 2, 3, 4}
  • Sacar 2 → vecinos 4(visitado), 5 → Cola = [3, 4, 5], Visitados = {0, 1, 2, 3, 4, 5}
  • Sacar 3 → vecino 5(visitado) → Cola = [4, 5]
  • Sacar 4 → vecino 5(visitado) → Cola = [5]
  • Sacar 5 → sin vecinos → Cola = []

Distancia de 0 a 5: 2 aristas (camino 0→2→5)

Java public void bfs(int start, int target) { Queue<Integer> q = new LinkedList<>(); boolean[] visitados = new boolean[V]; int[] dist = new int[V]; q.add(start); visitados[start] = true; dist[start] = 0; while(!q.isEmpty()) { int u = q.poll(); if(u == target) { System.out.println("Distancia: " + dist[u]); return; } for(int v : adj.get(u)) { if(!visitados[v]) { visitados[v] = true; dist[v] = dist[u] + 1; q.add(v); } } } }

Componentes Conexas con DFS

Dado el siguiente grafo no dirigido, encuentra todas las componentes conexas usando DFS:

Nodos: {0, 1, 2, 3, 4, 5, 6, 7} Aristas: 0-1, 0-2, 1-2, 3-4, 5-6, 5-7, 6-7 Visualmente: Componente A Componente B Componente C 0 --- 1 3 --- 4 5 --- 6 | / | / 2 7

Pregunta: ¿Cuántas componentes hay? ¿Cuáles son los nodos de cada una?

✅ Solución

3 componentes conexas:

  • Componente 1: {0, 1, 2} — DFS desde 0: visita 0→1→2
  • Componente 2: {3, 4} — DFS desde 3: visita 3→4
  • Componente 3: {5, 6, 7} — DFS desde 5: visita 5→6→7

Algoritmo: Para encontrar componentes, iterar sobre todos los nodos. Si un nodo no ha sido visitado, lanzar DFS desde él → todos los nodos visitados en ese DFS forman una componente.

Java public int contarComponentes() { boolean[] visitados = new boolean[V]; int componentes = 0; for (int i = 0; i < V; i++) { if (!visitados[i]) { componentes++; dfs(i, visitados); // Visita todo el componente } } return componentes; } private void dfs(int u, boolean[] visitados) { visitados[u] = true; for (int v : adj.get(u)) { if (!visitados[v]) dfs(v, visitados); } }

Ejercicio 3 — Dijkstra

★★★★

Camino más corto con Dijkstra

Aplica Dijkstra desde el nodo A en el siguiente grafo ponderado:

A --4-- B --2-- D | | / | 2 1 3 6 | | / | C --5-- E --1-- F Aristas: A→B(4), A→C(2), B→D(2), B→E(1), C→E(5), D→F(6), E→D(3), E→F(1)

Pregunta: ¿Cuál es la distancia mínima de A a cada nodo? ¿Cuál es el camino más corto de A a F?

✅ Solución

Distancias mínimas desde A:

  • A → A: 0
  • A → B: 4 (A→B)
  • A → C: 2 (A→C)
  • A → D: 6 (A→B→D, 4+2=6)
  • A → E: 5 (A→B→E, 4+1=5)
  • A → F: 6 (A→B→E→F, 4+1+1=6)

Camino más corto A→F: A → B → E → F (costo: 6)

Nota: A→C→E→F (2+5+1=8) es más largo aunque A→C es la arista más barata desde A

Java public void dijkstra(int start) { int[] dist = new int[V]; Arrays.fill(dist, Integer.MAX_VALUE); dist[start] = 0; PriorityQueue<int[]> pq = new PriorityQueue<>((a, b) -> a[1] - b[1]); pq.add(new int[]{start, 0}); while (!pq.isEmpty()) { int[] actual = pq.poll(); int u = actual[0], d = actual[1]; if (d > dist[u]) continue; // Ya encontramos un camino mejor antes for (Edge e : adj.get(u)) { if (dist[u] + e.weight < dist[e.target]) { dist[e.target] = dist[u] + e.weight; pq.add(new int[]{e.target, dist[e.target]}); } } } }

Ejercicio 4 — Bellman-Ford

★★★★

Detectar Ciclo Negativo

Aplica Bellman-Ford desde el nodo 0. ¿Hay un ciclo de peso negativo?

Nodos: {0, 1, 2, 3} Aristas: 0→1 (1), 1→2 (3), 2→3 (2), 3→1 (-7) Visualmente: 0 --1-→ 1 --3-→ 2 ↑ | | -7 2 | | 3 ←-----+ Ciclo: 1 → 2 → 3 → 1 (peso: 3 + 2 + (-7) = -2)

✅ Solución

¡SÍ hay un ciclo negativo! El ciclo 1→2→3→1 tiene peso total = 3 + 2 + (-7) = -2

Detección: Después de V-1 = 3 iteraciones de Bellman-Ford, hacemos una iteración extra. Si alguna arista aún puede relajarse, hay ciclo negativo.

  • Iteración 1: dist = [0, 1, ∞, ∞] → [0, 1, 4, ∞]
  • Iteración 2: dist = [0, 1, 4, 6] (relajamos 2→3)
  • Iteración 3: dist = [0, -1, 4, 6] (3→1: 6-7=-1 < 1 ✓)
  • Iteración extra: 1→2 se relaja (-1+3=2 < 4) → ¡CICLO NEGATIVO!
Java public boolean tieneCicloNegativo(int start) { int[] dist = new int[V]; Arrays.fill(dist, Integer.MAX_VALUE); dist[start] = 0; // Relajar todas las aristas V-1 veces for (int i = 1; i < V; i++) { for (Edge e : edges) { if (dist[e.source] != Integer.MAX_VALUE && dist[e.source] + e.weight < dist[e.target]) { dist[e.target] = dist[e.source] + e.weight; } } } // Iteración extra para detectar ciclos for (Edge e : edges) { if (dist[e.source] != Integer.MAX_VALUE && dist[e.source] + e.weight < dist[e.target]) { return true; // ¡Ciclo negativo encontrado! } } return false; }

Pregunta: ¿Cuál es la distancia mínima de 0 a 3? ¿Por qué camino?

✅ Solución

Matriz final:

0 1 2 3 0 [ 0, 5, 8, 9 ] 1 [ ∞, 0, 3, 4 ] 2 [ ∞, ∞, 0, 1 ] 3 [ ∞, ∞, ∞, 0 ]

Distancia 0→3 = 9 (camino: 0→1→2→3, costo 5+3+1=9)

Ordenamiento Topológico con DFS

Determina un orden topológico del siguiente DAG (Grafo Dirigido Acíclico) que representa dependencias de compilación:

Módulos: {main, parser, lexer, ast, codegen, utils} Dependencias (A→B significa "A depende de B"): main → parser, codegen parser → lexer, ast codegen → ast lexer → utils ast → utils ¿En qué orden debemos compilar para que cada módulo se compile DESPUÉS de sus dependencias?

✅ Solución

Orden topológico (una solución posible):

utils → lexer → ast → parser → codegen → main

Algoritmo DFS con pila: Al terminar de explorar un nodo (todos sus vecinos visitados), push a una pila. El resultado es la pila invertida.

  • DFS desde main: explora parser→lexer→utils(push)→lexer(push)→ast→utils(visitado)→ast(push)→parser(push)→codegen→ast(visitado)→codegen(push)→main(push)
  • Pila (top→bottom): main, codegen, parser, ast, lexer, utils
  • Invertir: utils, lexer, ast, parser, codegen, main ✓

Se agrega al terminar sus dependencias }

Ejercicio 7 — Dijkstra con Predecesores

★★★★

Reconstruir el Camino Más Corto

Modifica Dijkstra para reconstruir el camino, no solo la distancia. Dado este grafo, encuentra el camino más corto de 0 a 4:

Aristas: 0→1(2), 0→2(6), 1→3(5), 2→3(1), 3→4(2), 2→4(8) Agrega un arreglo prev[] donde prev[v] = nodo anterior en el camino más corto hacia v. Al relajar una arista u→v: if dist[u] + w < dist[v]: dist[v] = dist[u] + w prev[v] = u // ← Registrar predecesor

✅ Solución

Distancias y predecesores:

  • dist[0]=0, prev[0]=null
  • dist[1]=2, prev[1]=0 (0→1, costo 2)
  • dist[2]=6, prev[2]=0 (0→2, costo 6)
  • dist[3]=7, prev[3]=1 (0→1→3, costo 2+5=7. Empata con 0→2→3 = 6+1=7, pero el nodo 1 se procesa primero y la relajación usa < estricto: el empate NO actualiza prev)
  • dist[4]=9, prev[4]=3 (0→1→3→4, costo 7+2=9)

Camino 0→4: Reconstruir desde prev[]: 4←3←1←0

0 → 1 → 3 → 4 (costo total: 9)

Nota: 0→2→3→4 también cuesta 9 — ambos son caminos mínimos válidos; prev[] guarda el que el algoritmo encontró primero.

Java // Asumiendo que Dijkstra ya llenó el arreglo prev[] public List<Integer> reconstruirCamino(int[] prev, int destino) { List<Integer> camino = new ArrayList<>(); int actual = destino; // Rastrear hacia atrás desde el destino while (actual != -1) { // Asumiendo que prev[start] es -1 o null camino.add(actual); actual = prev[actual]; } // El camino se construyó al revés, hay que invertirlo Collections.reverse(camino); return camino; }

Ejercicio 8 — Dijkstra vs Floyd-Warshall

★★★★

Comparación Práctica

Dado el grafo, obtén las distancias mínimas entre TODOS los pares usando ambos algoritmos:

Nodos: {0, 1, 2} Aristas: 0→1(3), 0→2(8), 1→2(2), 2→0(5) Método 1: Ejecutar Dijkstra desde CADA nodo (3 veces) Método 2: Ejecutar Floyd-Warshall una vez ¿Cuál método es más eficiente para este grafo? ¿Y si el grafo tuviera 1000 nodos y 2000 aristas?

✅ Solución

Matriz de distancias (ambos dan el mismo resultado):

0 1 2 0 [ 0, 3, 5 ] ← 0→2: vía 1 (3+2=5 < 8) 1 [ 7, 0, 2 ] ← 1→0: vía 2 (2+5=7) 2 [ 5, 8, 0 ] ← 2→1: vía 0 (5+3=8)

Análisis de eficiencia:

  • Grafo pequeño (V=3, E=3): Ambos son rápidos. Floyd-Warshall es más simple de implementar.
  • V=1000, E=2000 (grafo disperso):
  • Dijkstra × V: O(V × (V+E) log V) = O(1000 × 3000 × 10) ≈ 30M ops
  • Floyd-Warshall: O(V³) = O(1000³) = 1000M ops
  • Conclusión: Para grafos dispersos grandes, Dijkstra × V es mejor. Floyd-Warshall es mejor para grafos densos (E ≈ V²).

Ejercicio 9 — Union Naive

★★★★

Trazar Union-Find con Union Naive (sin rank)

Dado un conjunto de 6 elementos {0, 1, 2, 3, 4, 5}, ejecuta las siguientes operaciones con union naive (siempre cuelga la raíz de x debajo de la raíz de y):

Estado inicial: parent = [0, 1, 2, 3, 4, 5] (cada elemento es su propia raíz) Operaciones: 1. union(1, 0) 2. union(3, 2) 3. union(5, 4) 4. union(3, 0) 5. union(5, 0) Regla union naive: parent[find(x)] = find(y) (la raíz de x se cuelga de la raíz de y)

Pregunta: ¿Cómo queda parent[] después de cada operación? ¿Cuántos saltos necesita find(5) al final (sin path compression)?

Traza paso a paso:

Inicio: parent = [0, 1, 2, 3, 4, 5] Conjuntos: {0} {1} {2} {3} {4} {5} 1. union(1, 0): find(1)=1, find(0)=0 → parent[1] = 0 parent = [0, 0, 2, 3, 4, 5] Conjuntos: {0,1} {2} {3} {4} {5} Árbol: 0←1 2. union(3, 2): find(3)=3, find(2)=2 → parent[3] = 2 parent = [0, 0, 2, 2, 4, 5] Conjuntos: {0,1} {2,3} {4} {5} Árbol: 2←3 3. union(5, 4): find(5)=5, find(4)=4 → parent[5] = 4 parent = [0, 0, 2, 2, 4, 4] Conjuntos: {0,1} {2,3} {4,5} Árbol: 4←5 4. union(3, 0): find(3)=2, find(0)=0 → parent[2] = 0 parent = [0, 0, 0, 2, 4, 4] Conjuntos: {0,1,2,3} {4,5} Árbol: 0←1, 0←2←3 5. union(5, 0): find(5)=4, find(0)=0 → parent[4] = 0 parent = [0, 0, 0, 2, 0, 4] Conjuntos: {0,1,2,3,4,5} Árbol: 0←1, 0←2←3, 0←4←5

find(5) sin path compression: 5→4→0 = 2 saltos

find(3) sin path compression: 3→2→0 = 2 saltos

⚠️ Con union naive, la altura depende del orden de las operaciones. En el peor caso (uniones encadenadas), la altura puede ser O(n).

Ejercicio 10 — Union by Rank

★★★★

Trazar Union-Find con Union by Rank

Mismas operaciones que el ejercicio anterior, pero ahora con union by rank. Traza parent[] y rank[] en cada paso:

Estado inicial: parent = [0, 1, 2, 3, 4, 5] rank = [0, 0, 0, 0, 0, 0] Operaciones (mismas que antes): 1. union(1, 0) 4. union(3, 0) 2. union(3, 2) 5. union(5, 0) 3. union(5, 4) Regla union by rank: Si rank[rx] < rank[ry]: parent[rx] = ry Si rank[rx] > rank[ry]: parent[ry] = rx Si rank[rx] == rank[ry]: parent[ry] = rx, rank[rx]++

Pregunta: ¿Cómo quedan parent[] y rank[]? ¿Cuántos saltos necesita find(4) sin path compression? Compara con la versión naive.

✅ Solución

Traza paso a paso:

Inicio: parent = [0, 1, 2, 3, 4, 5] rank = [0, 0, 0, 0, 0, 0] 1. union(1, 0): find(1)=1, find(0)=0 → rank iguales (0==0) → parent[0] = 1, rank[1]++ parent = [1, 1, 2, 3, 4, 5] rank = [0, 1, 0, 0, 0, 0] Árbol: 1←0 (1 es raíz, rank 1) 2. union(3, 2): find(3)=3, find(2)=2 → rank iguales (0==0) → parent[2] = 3, rank[3]++ parent = [1, 1, 3, 3, 4, 5] rank = [0, 1, 0, 1, 0, 0] Árbol: 3←2 (3 es raíz, rank 1) 3. union(5, 4): find(5)=5, find(4)=4 → rank iguales (0==0) → parent[4] = 5, rank[5]++ parent = [1, 1, 3, 3, 5, 5] rank = [0, 1, 0, 1, 0, 1] Árbol: 5←4 (5 es raíz, rank 1) 4. union(3, 0): find(3)=3, find(0)=1 → rank iguales (1==1) → parent[1] = 3, rank[3]++ parent = [1, 3, 3, 3, 5, 5] rank = [0, 1, 0, 2, 0, 1] Árbol: 3 (rank 2) / | \ 1 2 (3 tiene 3 hijos directos) | 0 5. union(5, 0): find(5)=5, find(0)=3 → rank[5]=1 < rank[3]=2 → parent[5] = 3 (5 se cuelga de 3) parent = [1, 3, 3, 3, 5, 3] rank = [0, 1, 0, 2, 0, 1] Árbol: 3 (rank 2) / | \ \ 1 2 5 | | 0 4

find(4) sin PC: 4→5→3 = 2 saltos (mismo que naive aquí)

Diferencia clave: La raíz con union by rank es 3 (rank 2), mientras que con naive es 0. El árbol con rank es más ancho y plano — la altura máxima está garantizada ≤ log₂(6) ≈ 2.58, es decir, máximo 2.

✅ Con union by rank + path compression, cada operación tarda O(α(n)) ≈ O(1). ¡La combinación perfecta para Kruskal!

Ejercicio 11 — Kruskal con Union-Find

★★★★

Trazar Kruskal usando Union-Find

Aplica Kruskal al siguiente grafo no dirigido (el mismo de la animación). Usa union naive y traza parent[] en cada paso:

Nodos: {0, 1, 2, 3} Aristas: 2-3 (4), 0-3 (5), 0-2 (6), 0-1 (10), 1-3 (15) 0 / | \ 10 5 6 / | \ 1 3----2 \ | 4 15 | \ | (1-3) Estado inicial: parent = [0, 1, 2, 3] Regla: union(u, v) → parent[find(u)] = find(v)

Pregunta: ¿En qué orden se evalúan las aristas? ¿Cuál se descarta y por qué? ¿Cuál es el costo total del MST y cómo queda parent[]?

✅ Solución

Paso 1: ordenar aristas por peso ascendente: (2-3):4, (0-3):5, (0-2):6, (0-1):10, (1-3):15

Traza:

Inicio: parent = [0, 1, 2, 3] 1. Arista 2-3 (peso 4): find(2)=2, find(3)=3 → distintos → UNIR parent[2] = 3 → parent = [0, 1, 3, 3] MST = {2-3} costo = 4 2. Arista 0-3 (peso 5): find(0)=0, find(3)=3 → distintos → UNIR parent[0] = 3 → parent = [3, 1, 3, 3] MST = {2-3, 0-3} costo = 9 3. Arista 0-2 (peso 6): find(0)=3, find(2)=3 → ¡MISMA RAÍZ! → DESCARTADA: formaría el ciclo 0-3-2-0 4. Arista 0-1 (peso 10): find(0)=3, find(1)=1 → distintos → UNIR parent[3] = 1 → parent = [3, 1, 3, 1] MST = {2-3, 0-3, 0-1} costo = 19 MST completo: |MST| = 3 = V-1 → PARAR (la arista 1-3 de peso 15 ni se evalúa)

Costo total del MST = 19 · Aristas: {2-3, 0-3, 0-1} · parent final = [3, 1, 3, 1]

💡 Aquí se ve el papel de Union-Find: en el paso 3, find(0) == find(2) detecta el ciclo en tiempo casi O(1), sin recorrer el grafo.
